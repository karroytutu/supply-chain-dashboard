# 项目开发规范文档（rules.md）

版本：v1.1
适用范围：本项目所有开发、测试、部署相关操作（含AI辅助开发）
核心原则：规范统一、环境隔离、可追溯、自动化落地，贴合单仓库+dev/prod分离结构

---

> ⚠️ **【最高优先级警告】代码修改前必须检查分支！**
> 
> **任何代码修改操作前，AI 必须首先执行分支检查，禁止跳过！**
> 
> 检查命令：`git branch --show-current`
> - 若返回 `main`：必须先创建功能分支，禁止在 main 上直接修改代码
> - 若返回其他分支：可继续开发
> 
> **违反此规则视为严重违规！**

## 一、项目结构规范（必严格遵循）

项目根目录结构固定，禁止随意新增/删除一级目录，前后端目录命名、功能严格区分：

```plain text
你的项目/
├── .git               # 唯一Git仓库，禁止嵌套.git
├── .gitignore         # Git忽略规则，禁止修改prod相关忽略配置
├── dev/               # 开发环境（仅开发使用，提交至Git）
│   ├── frontend/      # 前端开发目录（如Vue/React项目）
│   └── backend/       # 后端开发目录（如Node/Java项目）
└── prod/              # 生产环境（仅服务器部署使用，不提交至Git）
    ├── frontend/      # 前端生产打包产物
    └── backend/       # 后端生产部署文件
```

- dev目录：仅存放开发阶段的代码、开发配置，所有开发操作均在dev下进行

- prod目录：仅在服务器本地存在，用于部署生产环境，禁止在本地dev环境操作prod目录

- 前后端目录：各自独立，前端仅在dev/frontend开发，后端仅在dev/backend开发，禁止跨目录写代码

## 二、Git分支规范（AI辅助执行，必遵循）

### 2.0 强制第一步（零容忍规则）

**当用户提出代码修改需求时，AI 必须首先执行以下操作：**

1. **第一句话必须说**："我来检查当前分支状态"
2. **第一个操作必须是**：执行 `git branch --show-current` 和 `git status`
3. **禁止跳过此步骤**：无论任务多么紧急，都必须先检查分支

```
# 强制第一步命令（必须在任何代码修改前执行）
git branch --show-current   # 查看当前分支
git status                  # 查看工作区状态
```

**违规判定**：如果 AI 在没有执行分支检查的情况下直接修改代码，即视为违规。

### 2.1 分支命名规范

所有分支命名统一格式，AI操作时需严格按此命名，禁止随意命名：

- 主分支：main（唯一生产分支，仅用于合并、部署，禁止直接在main上开发/修改）

- 功能分支：feature/xxx（xxx为功能描述，小写英文+连字符，如feature/login、feature/order-list）

- 修复分支：fix/xxx（xxx为修复描述，如fix/login-error、fix/backend-interface）

- 临时分支：temp/xxx（仅用于临时测试，测试完成后立即删除，禁止合并至main）

### 2.2 分支操作流程（AI执行核心步骤）

**【强制前置检查】**

在执行任何代码修改操作前，AI必须先完成以下检查：

1. **检查当前分支**：执行 `git branch --show-current`，确认当前不在 main 分支
2. **检查工作区状态**：执行 `git status`，确认工作区干净或了解未提交的修改
3. **若当前在 main 分支**：必须先创建功能分支，禁止在 main 上直接修改代码

```
# 前置检查命令示例
git branch --show-current   # 查看当前分支
git status                  # 查看工作区状态
```

完整的开发流程如下：

**1. 确保main分支为最新状态**
- 切换到main分支，确保基于最新代码开发
- 执行：git checkout main
- 确保工作区干净，无未提交的修改

**2. 创建功能分支（本地）**
- 按规范命名分支（如feature/login、feature/order-list）
- 执行：git checkout -b feature/xxx

**3. 同时开发前端 + 后端（本地dev目录）**
- 前后端在同一个功能分支中开发，保持功能完整性
- 前端代码提交至dev/frontend，后端代码提交至dev/backend

**4. 自测（本地环境，前端连后端本地服务）**
- 启动后端本地服务
- 启动前端开发服务器，连接后端本地接口
- 确认功能正常后再进行下一步

**5. 代码审查与提交**
- 开发完成后进行代码审查，检查代码规范、bug、安全问题
- 确认无误后执行提交：git add . → git commit -m "feat: xxx"
- 所有代码提交至对应分支，禁止直接提交至main分支

**6. 合并到main分支**
- 代码审查通过后，切换到main分支：git checkout main
- 合并功能分支：git merge feature/xxx
- 合并后删除原功能分支：git branch -d feature/xxx，保持分支整洁

**7. 部署到生产环境**
- main分支合并后，执行部署流程：
  - 构建前端（npm run build → 生成dist）
  - 构建后端（编译TypeScript → 生成生产代码）
  - 分别部署前端/后端容器或服务
- 部署完成后校验服务可用性

### 2.3 禁止操作

- 禁止直接在main分支写代码、提交修改

- 禁止创建无意义分支、命名不规范分支

- 禁止合并未经过代码审查的功能分支

- 禁止跳过自测直接合并代码

## 三、开发规范（前后端通用+各自补充）

### 3.1 通用开发规范

- 代码风格：统一代码格式（如前端ESLint、后端代码规范），AI开发时需自动遵循对应规范，避免格式混乱

- 注释规范：关键代码（如接口、核心逻辑、工具函数）必须加注释，清晰说明功能、参数、返回值，便于后期维护

- 文件命名：小写英文+连字符，禁止中文、大写、特殊字符，如frontend/src/components/login-form.vue、backend/src/utils/request.js

- 依赖管理：前后端依赖各自在对应目录的package.json（或对应配置文件）中管理，禁止跨目录引用依赖

- 自测要求：开发完成后，必须在dev环境自测（前端调后端本地接口），确认功能正常后再合并到main分支

### 3.2 前端开发规范（dev/frontend）

- 目录结构：按框架规范组织（如Vue：src/components、src/views、src/api、src/utils），禁止随意新增一级目录

- 接口调用：统一封装请求工具，接口地址、请求方式集中管理，禁止在组件内直接写接口地址

- 样式管理：统一使用一种样式方案（如Scoped CSS、Tailwind），禁止混用，避免样式冲突

- 打包规范：开发完成后，执行npm run build生成dist文件，仅用于prod部署，禁止提交dist至Git（.gitignore已配置）

### 3.3 后端开发规范（dev/backend）

- 目录结构：按业务模块组织（如src/controller、src/service、src/model、src/config），禁止随意新增一级目录

- 接口规范：统一接口返回格式（如{code: 200, message: "success", data: {}}），异常处理统一封装

- 配置管理：开发环境配置放在dev/backend/.env（或对应配置文件），禁止硬编码配置（如数据库地址、密钥）

- 日志管理：统一日志输出格式，日志文件放在dev/backend/logs，禁止提交日志至Git（.gitignore已配置）

## 四、验证步骤编写规范

在编写代码修改的验证步骤时，必须先确认服务的运行状态：

1. **不要假设服务未启动**：验证步骤不应默认包含启动服务的命令
2. **正确写法**：
   - 如果服务已运行：写"刷新页面即可看到修改"或"确保服务运行中"
   - 如果不确定：写"确保前端/后端服务运行中（端口 xxx）"
3. **错误写法**：直接写"启动前端服务"、"启动后端服务"

### 示例对比

**错误**（假设服务未启动）：
```
1. 启动后端服务：cd dev/backend && npm run dev
2. 启动前端服务：cd dev/frontend && npm run dev
3. 验证功能...
```

**正确**（先确认状态）：
```
1. 确保后端服务运行中（端口 8100），若未启动：cd dev/backend && npm run dev
2. 确保前端服务已启动（端口 3100），刷新页面即可看到修改，若未启动：cd dev/frontend && npm run dev
3. 验证功能...
```

### 端口参考（supply-chain-dashboard项目）
- 开发环境前端：3100
- 开发环境后端：8100
- 生产环境前端：3000
- 生产环境后端：8000

## 五、提交规范（AI自动执行，必遵循）

### 5.1 提交信息格式

所有git commit信息统一格式，AI提交时需严格按此填写，便于追溯：

```plain text
类型: 描述（简洁明了，不超过50字）
```

类型说明：

- feat：新增功能（如feat: 新增登录功能）

- fix：修复bug（如fix: 修复登录密码加密异常）

- docs：修改文档（如docs: 更新开发规范）

- style：代码格式调整（不影响功能，如style: 格式化代码）

- refactor：重构代码（不新增功能、不修复bug，如refactor: 优化接口逻辑）

- test：新增/修改测试用例（如test: 新增登录接口测试）

### 5.2 提交要求

- 每次提交仅对应一个功能/一个bug修复，禁止一次提交多个不相关修改

- 提交前需确认代码可运行、无语法错误，AI需自动检查后再提交

- 禁止提交无用文件（如node_modules、日志、临时文件），.gitignore已配置，AI需严格遵循

## 六、代码审查规范

1. 审查时机：开发完成后，在合并到main分支前必须进行代码审查

2. 审查内容：检查代码规范、bug、安全问题、注释完整性

3. 审查流程：
   - 开发者完成功能开发并自测通过
   - 进行代码审查，可自行检查或由团队成员协助审查
   - 发现问题则修改代码，重新提交
   - 审查通过后方可合并到main分支

4. 异常处理：若审查发现问题，需及时修改代码，重新提交直至审查通过

## 七、部署规范

- 环境隔离：dev环境用于开发、自测，prod环境用于生产部署，禁止在prod环境直接修改代码、配置

- 部署流程：main分支合并后，执行部署操作
  - 构建前端：cd dev/frontend && npm run build，产物输出到 prod/frontend/dist/
  - 构建后端：cd dev/backend && npm run build，产物输出到 prod/backend/dist/
  - 部署服务：使用Docker或直接运行生产环境服务

- 部署要求：部署前需确保所有测试通过，部署完成后校验服务可用性

- 回滚规范：若部署出现问题，通过Git回退到上一个稳定版本，重新执行部署流程

## 八、AI辅助开发规范

- **【强制】分支检查优先**：AI 收到代码修改需求后，必须首先执行分支检查，这是零容忍规则，没有任何例外情况

- AI操作指令：所有AI操作需遵循本规范，指令需清晰明确（如"创建feature/order-list分支开发订单列表功能"）

- AI审查要求：AI需严格按本规范的开发、提交要求，对代码进行全面审查，重点检查分支规范、代码格式、注释、无无用文件

- 异常反馈：若AI操作不符合规范（如命名错误、提交信息不规范），需及时修正，重新执行操作

## 九、违规处理

- 若出现分支命名不规范、提交信息不符合要求，需删除对应提交/分支，重新操作

- 若误将prod目录提交至Git，需立即删除Git中的prod相关记录，重新配置.gitignore

- 若未经过代码审查直接合并到main分支，需立即回滚合并操作，重新走代码审查流程

## 十、附则

1. 本规范自发布之日起执行，所有开发、AI操作均需严格遵循；
2. 若项目结构、技术栈变更，本规范将同步更新；
3. 若有疑问，按本规范优先执行，或补充规范说明。
