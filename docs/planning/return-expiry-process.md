# 临期退货处理流程 - 实施计划

## 一、背景与目标

### 业务背景
- 每天有大量退货，其中部分商品已临期需要责任到人处理
- 临期退货需要经过采购主管确认流程（按商品维度，确认一次后同商品不再重复确认）
- 采购主管确认可采购退货后，需要在规定时限内填写ERP退货单号（由采购主管填写）
- 不能采购退货的，进入营销限时销售流程（无明确时限，只需在过期前完成）
- 全程需要钉钉消息推送提醒

### 核心流程
```
退货入库 → 数据同步 → [商品退货规则判断] → 规则为可退货? → Yes → 采购主管填写ERP退货单 → 仓储执行退货
                              ↓ No
                       进入采购确认 → 采购主管确认 → Yes → 同上流程
                                              ↓ No
                                       营销师销售 → 完成（在过期前完成即可）
```

### 临期品定义
临期品是指商品剩余保质期低于阈值的商品，阈值根据商品保质期（shelfLife）动态计算：

| 商品保质期（天） | 临期阈值（天） | 说明 |
|-----------------|---------------|------|
| ≤90 | 30 | 短保商品，阈值30天 |
| 91-150 | 45 | 中保商品，阈值45天 |
| 151-270 | 60 | 中长保商品，阈值60天 |
| ≥271 | 90 | 长保商品，阈值90天 |

**判定逻辑**：当 `商品当前剩余保质期 ≤ 临期阈值` 时，该商品被判定为临期品。

### 推送机制
1. **每天08:30**：新增临期退货提醒（推送对象：采购主管）
2. **每天08:30**：待填写ERP退货单提醒（推送对象：采购主管）
3. **采购确认为不可退货时**：无法采购退货通知（推送对象：责任营销师）
4. **采购填写ERP退货单后**：待仓储退货通知（推送对象：仓储主管）

## 三、数据模型

### 核心表结构

| 表名 | 用途 |
|------|------|
| `expiring_return_orders` | 临期退货单主表 |
| `expiring_return_actions` | 退货流程操作记录表 |
| `goods_return_rules` | 商品退货规则表（按商品维度记录是否可采购退货） |
| `return_flow_configs` | 流程配置表（时限配置） |
| `dingtalk_msg_templates` | 钉钉消息模板表 |

### 商品退货规则表（新增）

此表用于记录商品是否可以采购退货，确认一次后同商品不再重复确认。

```sql
CREATE TABLE goods_return_rules (
  id SERIAL PRIMARY KEY,
  goods_id VARCHAR(64) NOT NULL,                    -- 商品ID
  goods_name VARCHAR(200) NOT NULL,                -- 商品名称
  can_return_to_supplier BOOLEAN NOT NULL,         -- 是否可以采购退货
  confirmed_by INTEGER REFERENCES users(id),          -- 确认人
  confirmed_at TIMESTAMP,                           -- 确认时间
  comment TEXT,                                     -- 备注说明
  is_active BOOLEAN DEFAULT TRUE,                  -- 是否生效
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(goods_id, is_active)                      -- 每个商品只有一个生效规则
);
```

### 退货单状态流转

| 状态代码 | 显示名称 | 说明 |
|----------|----------|------|
| `pending_erp_fill` | 待填ERP退货单 | 待填写ERP采购退货单号（由采购主管操作） |
| `pending_warehouse_execute` | 待仓储退货 | 待仓储主管执行退货（需录入退货情况） |
| `pending_marketing_sale` | 待营销销售 | 待责任营销师完成限时销售（无明确时限） |
| `completed` | 已完成 | 流程结束 |
| `cancelled` | 已取消 | 已取消 |

**注意**：
- 采购确认环节按商品维度处理，商品确认规则保存在 `goods_return_rules` 表
- 新退货入库时，系统自动根据商品退货规则判断进入哪个流程
- 营销销售无明确时限，只需在商品过期前完成
- 仓储主管在执行退货时需要在系统中录入退货情况（退货数量、退货时间等）

### 关键配置项

| 配置键 | 默认值 | 说明 |
|--------|--------|------|
| （暂无特殊配置项） | - | 各环节无明确时限要求 |

## 四、API 设计

```
退货单管理:
- GET    /api/return-orders                # 退货单列表
- GET    /api/return-orders/:id            # 退货单详情
- POST   /api/return-orders/:id/erp-fill  # 填写ERP退货单
- POST   /api/return-orders/:id/warehouse  # 仓储执行
- POST   /api/return-orders/:id/marketing  # 营销销售完成
- GET    /api/return-orders/pending-erp    # 待填写ERP退货单的退货单

商品退货规则管理:
- GET    /api/goods-return-rules           # 规则列表
- POST   /api/goods-return-rules           # 创建规则
- PUT    /api/goods-return-rules/:id       # 更新规则
- POST   /api/goods-return-rules/batch     # 批量设置规则
- GET    /api/goods-return-rules/check/:goodsId  # 检查商品是否有规则

配置管理:
- GET    /api/return-configs               # 获取配置
- PUT    /api/return-configs/:key           # 更新配置
```

## 五、推送机制设计

### 统一推送时间
- **所有推送时间**: 每天早上 8:30

### 推送类型

| 推送类型 | 触发条件 | 推送对象 | 内容 |
|----------|----------|----------|------|
| 新增临期退货提醒 | 每天08:30同步后 | 采购主管 | 今日新增临期退货数量和基本信息 |
| 待填ERP退货单提醒 | 每天08:30检测 | 采购主管 | 尚未填写ERP退货单号的退货单列表 |
| 无法采购退货通知 | 采购主管确认为不可退货时 | 责任营销师 | 商品无法采购退货，需寻找渠道销售 |
| 待仓储退货通知 | 采购填写ERP退货单后 | 仓储主管 | 需尽快安排商品退出，并录入退货情况 |

## 十一、数据来源与关联关系

### 关键业务表

| 表名 | 数据库 | 用途 |
|------|--------|------|
| `云仓退货验收明细` | xinshutong | 临期退货数据来源 |
| `客户档案表` | xinshutong | 获取营销师信息 |
| `销售结算明细表` | xinshutong | 关联客户信息 |
| `商品档案` | xinshutong | 获取商品保质期信息 |

### 责任人关联逻辑

**关联公式**: `云仓退货验收明细.sourceBillNo = 销售结算明细表.originStr`

```
责任营销师关联流程：
  步骤1: 从【云仓退货验收明细】（数据库xinshutong）获取 sourceBillNo（退货单号）
  步骤2: 在【销售结算明细表】（数据库xinshutong）中通过 originStr = sourceBillNo 匹配
  步骤3: 获取 consumerName（客户名称）
  步骤4: 在【客户档案表】（数据库xinshutong）中通过 name = consumerName 匹配
  步骤5: 获取 consumerManagerName（责任营销师）
```

**涉及数据表说明**:

| 数据表 | 所属数据库 | 字段 | 说明 |
|--------|-----------|------|------|
| 云仓退货验收明细 | xinshutong | sourceBillNo | 退货单号，作为关联起点 |
| 销售结算明细表 | xinshutong | originStr | 来源单号，用于匹配sourceBillNo |
| 销售结算明细表 | xinshutong | consumerName | 客户名称 |
| 客户档案表 | xinshutong | name | 客户名称 |
| 客户档案表 | xinshutong | consumerManagerName | 责任营销师姓名 |

### 退货数据同步机制

#### 定时任务配置
| 配置项 | 值 |
|--------|------|
| 任务名称 | sync_return_orders |
| 执行时间 | 每天 08:30:00 |
| 时区 | Asia/Shanghai (UTC+8) |
| cron表达式 | `0 30 8 * * *` |

#### 同步逻辑
```
1. 查询【云仓退货验收明细】（数据库xinshutong）昨天（00:00:00 ~ 23:59:59）新增的记录

2. 对于每条退货记录：
   2.1 查询【商品档案】（数据库xinshutong）获取 shelfLife（保质期）
   2.2 根据临期品定义计算阈值，判断是否临期
   2.3 仅处理临期商品

   2.4 查询商品退货规则（goods_return_rules）：
       - 若存在生效规则 → 根据规则判断流程
       - 若不存在规则 → 标记待确认（不自动进入流程）

   2.5 关联责任人（按上述责任人关联逻辑）

   2.6 创建临期退货单记录

3. 同步完成后：
   3.1 发送钉钉通知给采购主管（新增临期退货）
```
